DBMON-1593 · Option B

Receiver Tags
via Context

How we inject pipeline metadata into context.Context at the receiver layer — without touching pdata — and make it available to downstream processors.

opentelemetry-collector consumer/tagsconsumer client.Info.Metadata
01 · PROBLEM

The Gap

  • Receivers have no standard way to attach pipeline-level metadata to the telemetry they produce
  • Processors like resourceprocessor can only read resource attributes — they cannot read receiver config
  • Users want to tag data at ingest: env=prod, team=platform
  • Writing to pdata inside the receiver would mutate shared data — unsafe in fan-out pipelines
# What users want to write: receivers: otlp/prod: tags: env: prod team: platform # And have a processor pick it up: processors: resource/add-tags: attributes: - key: deployment.env from_context: metadata.tags.env action: upsert
02 · SOLUTION OVERVIEW

Tags travel on context.Context — pdata is never touched.

04 · NEW PACKAGE

consumer/tagsconsumer

// context.go — dual injection func ContextWithTags( ctx context.Context, tags map[string]string, ) context.Context { // 1. custom key for clean internal API ctx = context.WithValue(ctx, tagsKey{}, tags) // 2. merge into client.Info.Metadata // prefix "tags." avoids collisions ci := client.FromContext(ctx) merged := make(map[string][]string) for k := range ci.Metadata.Keys() { merged[k] = ci.Metadata.Get(k) } for k, v := range tags { merged["tags."+k] = []string{v} } ci.Metadata = client.NewMetadata(merged) return client.NewContext(ctx, ci) }
Wrapper pattern
  • Implements consumer.Logs/Metrics/Traces
  • Calls ContextWithTags then delegates to next
  • Capabilities() delegates — no MutatesData claim
Fast path
if len(tags) == 0 { return next // zero overhead }

Receivers with no tags skip the wrapper entirely

05 · CONFIG UNMARSHALING

Stripping tags: Before Factory Sees It

confmap uses ErrorUnused: true — an unknown key like tags: causes a fatal error unless we strip it first.

// configs.go — NewReceiverConfigs sets // extractTags: true (receivers only) for id, sub := range rawCfgs { rawMap := sub.ToStringMap() // pull tags out before unmarshal if c.extractTags { if t, ok := rawMap["tags"]; ok { c.tags[id] = coerceToStringMap(t) delete(rawMap, "tags") } } filtered := confmap.NewFromStringMap(rawMap) // now safe to unmarshal into factory cfg factory.UnmarshalConfig(filtered, &cfg) }
Key invariant
  • NewReceiverConfigs → extractTags ON
  • NewConfigs → extractTags OFF
  • Processors/exporters still error on unknown tags:
Non-map value → hard error
tags: "string" returns:
"tags" must be a map of string keys to string values
06 · PROCESSOR COMPATIBILITY

Reading Tags Downstream

from_context: in attributesprocessor / resourceprocessor reads only from client.Info.

# Works today — no processor changes: processors: resource/tags: attributes: - key: deployment.environment from_context: metadata.tags.env action: upsert - key: team from_context: metadata.tags.team action: upsert
attraction.go:getAttributeValueFromContext looks up metadata.* via ci.Metadata.Get(key). Our tags.* keys land there automatically.
Why tags. prefix?
  • HTTP headers / gRPC metadata share the same Metadata map
  • Prefix isolates our keys from transport metadata
  • A tag named host → stored as tags.host
  • No risk of clobbering existing processor reads
Internal API still clean
  • tagsconsumer.FromContext(ctx) reads from tagsKey{}
  • No string prefix parsing needed
  • Type-safe map[string]string
07 · FUTURE PATH

Preferred Long-Term Approach

The metadata.tags.* indirection is a compatibility bridge. The cleaner end state requires one contrib change.

Today (bridge)
from_context: metadata.tags.env # Reads: ci.Metadata.Get("tags.env") # Works with zero contrib changes # But exposes metadata. indirection
Preferred (one contrib PR)
from_context: tags.env # attraction.go gets tags.* handler # reads tagsconsumer.FromContext(ctx) # clean separation of concerns
Blocked on
  • Field name agreement upstream — is tags the right name?
  • One PR to opentelemetry-collector-contrib: add tags.* prefix in attraction.go:getAttributeValueFromContext
  • Both mechanisms coexist — migration is non-breaking
